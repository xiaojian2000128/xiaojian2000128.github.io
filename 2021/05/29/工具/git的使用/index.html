git的介绍和使用

git概述

git是一个免费的,开源的分布式版本控制工具.相对集中式版本控制工具,git的版本控制能在本地进行.

git的工作机制

工作区(写代码)—>git add—>暂存区(临时存储)—>git commit—>本地库(历史版本)—>git push—>远程库

git和代码托管中心

代码托管中心是基于网络服务器的远程代码仓库.局域网:gitLab,互联网:gitHub(国外),gitee(国内)

基础

创建git仓库

克隆远程仓库到本地: git clone <url> [自定义本地仓库名]

将本地的某个文件夹纳入Git的管理: 在该文件夹下, 使用 git init

git跟踪文件

将文件纳入到git管理: git add 文件路径, 会将该文件放入到暂存区, 注意如果这个命令的路径指定范围包括了工作区且已修改的文件, 也会将其暂存

让某些文件不被git管理: 配置忽略文件, 在.git同级目录下创建或修改.gitignore文件, 可以让这些文件被Git管理

显示当前工作状态

查看暂存区, 工作区以及未跟踪的文件状态: git status [-s], 该命令也可以显示出当前所处的分支, -s可以用简洁方式显示, 左边是暂存区, 右边是工作区, A新跟踪的文件, M修改的文件, ??未跟踪的文件

git提交

暂存已修改的文件: git add 文件路径

提交暂存区中的文件: git commit [-a] [-m “注释”]. -a, 把工作区跟踪的文件和暂存区一起提交, 如果工作区和暂存区的同时存在该文件, 则以工作区为准

对上一提交进行修改, 例如修改提交注解, 和补交文件等等: git commit –amend, 还可以配合上-a, -m选项

git恢复

进行提交时, 不对某个文件进行提交: git restore –staged 文件路径, 也可以使用git reset HEAD 文件路径, 这个命令是使用HEAD指向的提交中的文件放入到暂存区中, git status中显示的区与区之间的差异

将某个文件恢复到上次暂存或提交的样子: git restore 文件路径, 也可以使用git checkout – 文件路径

差异对比

显示文件从上次暂存以来的改动: git diff [blob], blob是正则表达式的子集

显示暂存区文件自上次提交以来的改动: git diff –staged/–cached [blob]

查看某一次提交与它的上一次提交的差异: git show (分支名或哈希值), 与git log -p 显示的格式一致

从git管理中移除

跟踪了不需要的文件, 将其取消跟踪: git rm –cached [-f] <file>, 如果暂存区和工作区同时有该文件, 必须要加-f

删除Git管理文件: 方法一: git rm [-f] <file>, 如果文件在暂存未提交或修改为暂存时, 需要加-f; 方法二: 直接删除, 提交时加-a, 也可以在硬盘中和Git中同时删除该文件

重命名文件或移动文件: git mv <source> <destination>, 如果使用Git之外的方式, 还需要手动跟踪重命名或移动后的文件和暂存删除操作

查看提交历史

查看提交历史, 快速查找某次提交: git log [-p] [-n] [-S] [–stat]. -p 可以该次提交与上一次提交的改动; -n 显示的条数; –stat 显示出自上次提交以来每个文件修改了多少行, 有多少个文件被修改; 扩展选项, –after –after=”时间”, 时间的写法有两种, 其一: 1.day或2.weeks 其二: “yyyy-MM-dd”; –before; –author; –committer; –grep; -S -S “字符串”, 仅显示本地提交有增加或删除该字符串的提交

别名

使用别名简化某些git命令的输入: git config –global alias.自定义的别名 git命令, 写好的别名将会存储到全局配置文件中, git命令不需要带git, 同时如果有空格, 要是单引号(Unix)或双引号(Windows)将整个命令包在里面

远程仓库

远程仓库的管理

查看已有的远程仓库: git remote [-v]. -v 显示出该仓库的url地址

添加远程仓库: git remote add <远程仓库的名字> <url>, 远程仓库的名字不可以省略, 任何要使用url的使用都可以使用远程仓库名来使用

移除远程仓库: git remote remove 远程仓库名

对远程仓库重命名: git remote rename oldName newName, 可以对clone时默认的仓库名origin进行改名

查看某个远程仓库拥有的分支/查看本地分支与该仓库的分支的跟踪关系: git remote show 远程仓库名

使用远程仓库

从某个远程仓库中拉取数据但不更新: git fetch [远程仓库名], 如果省略仓库名, 默认是origin

从某个远程仓库中拉取数据同时更新当前跟踪分支: git pull [<远程仓库名>], 如果当前分支有跟踪的远程分支, 该分支会合并远程分支到当前分支, 默认是origin. 这个命令和上个命令一样会拉取所有的分支, 但不会创建本地跟踪分支. 要想让本地分支去跟踪远程分支, 需要拉取远程分支所在的仓库. git pull只会更新当前分支

推送某个分支到某个远程仓库: git push [远程仓库名] [本地分支名], 如果不指定远程仓库, 则默认是origin, 如果不指定本地分支则默认是当前分支

分支

分支管理

当做一个与当前分支无关的事时, 例如, 修以前的BUG, 修完合并该分支. 或做一些明确知道会遗弃的事, 不合并, 事后删除就行, 可以创建一个新分支, 在该分支上操作: git branch 新分支名

不需要该分支, 该分支被合并了或该分支只是用于test分支, 不需要进行合并: git branch -d 分支名, 如果该分支没有被合并过, 需要讲-d改为-D, 否则会提示 not fully merged

查看所有的本地分支, 查看分支的合并情况: git branch [-v] [–merged/–no-merged] [分支名], -v: 用于显示该分支的最后一次提交注释; –merged/–no-merged: 显示已经合并到该分支的和未合并到该分支的; 分子名: 用于指定分支, 不写意味者当前分支, 指定了, 就必须配合上–merged或–no-merged

分支切换

切换分支: git checkout -b 分支名, 工作目录会切换到该分支的最新提交. 切换分支时, 需要保证当前工作环境干净, 否则不允许切换. 这个问题可以通过暂存和修补提交(现在先提交, 以后对这次提交进行修改). 但是如果两个分支指向同一次提交, 工作目录中可以有未提交文件.

创建分支并切换分支: git checkout -b 新分支

分支合并

将某个分支合并到当前分支: git merge 要进行合并的分支. 合并有三种情况: 1. fast-forward: 如果要合并的分支是当前分支的直接后继, 则会显示fast-forward; 2. recursive: 如果两个分支在不同路径下, 且没有修改相同的文件, 会变成三方合并, 会生成一个新的提交, 生成时, 需要输入提交注释; conflict:如果两个分支不在同一路径下, 且修改了相同的文件(也许是相同文件的同一地方), 会变成conflict. 该情况, 会修改冲突文件, 你需要手动解决冲突, 步骤, 将冲突文件改成自己想要的样子, 然后git add 该文件就可以解决冲突了. 提交时, 会自动把要冲突的分支合并进来

标签

标签管理

对某次很重要的提交进行标记, 方便以后查找: git tag -a 标签名 [提交的hashi] -m “标签附注”, 如果不指定提交的hash, 默认为HEAD指向的提交

查看标签列表: git tag [-l blob字符串]

查看该标签所指向提交的具体信息: git show 标签名

删除本地标签: git tag -d 标签名

共享标签

将本地的标签传送到远程仓库上: git push 远程仓库名 [本地标签名|–tags], 默认情况下, git push不会传送标签到远程仓库上. –tags, 会传送本地的所有标签

删除共享标签: git push 远程仓库名 -delete 标签名

检出标签

会出现分离状态的检出标签: git checkout 标签名, 此时HEAD不指向任何分支, 直接指向提交

不会出现分离状态的检出标签: git checkout -b 新分支名 标签名

远程分支

远程分支管理

在远程仓库上创建分支: git push 远程仓库名 本地分支名, 远程分支和本地分支不会有跟踪关系

删除远程仓库上的分支: git push 远程仓库 –delete 远程分支名

拉取

更新本地仓库与远程仓库一致: git fetch 远程仓库, 会移动本地仓库中的远程仓库/分支名指针与远程仓库中的一致, 这个命令不会把远程分支合并进

推送

推送某个本地分支到某个仓库的远程分支: git push 远程仓库名 本地分支名[:远程分支名]. 如果省略远程分支名, 有相同名称的远程分支, 更新该分支; 没有相同名称的远程分支, 用本地分支名新建一支远程分支. 如果本地分支建立了跟踪分支, 可以省略本地分支名; 如果没有建立跟踪分支, 就必须要些本地分支名. 本质: 移动远程仓库/分支名指针到当前分支, 再发送到远程仓库

跟踪分支

创建与远程分支名称的跟踪分支: git checkout 远程分支名, 如果这里远程分支带了远程仓库名, 将会出现HEAD分离状态, 使用git reflog可以获得分离的提交

有多个远程仓库有相同的分支名, 创建跟踪分支: git checkout -t 远程仓库/远程分支名

创建自定义名称的跟踪分支: git checkout -b 自定义跟踪分支名 远程仓库/远程分支名

更改当前本地分支去跟踪另外一个远程分支: git branch -u 远程仓库/远程分支

查看跟踪的情况: git branch -vv, 这个命名是根据本地数据显示的, 没有和服务器同步. 要想使用同步的, 先git fetch –all, 再git branch -vv

变基

将两个分支合并成一条线: 1. git checkout 要被合并的分支; 2. git rebase 作为基底的分支; 3. git checkout 基底分支; 4. git merge 要被合并的分支. 简洁方式: git rebase 基底分支 要被合并的分支, git merge 要被合并的分支. 变基的本质是将两个分支的差异取出, 然后放到基底啊分支上. 只在本地仓库进行变基

多分支进行变基: git rebase –onto 基底分支 需要进行排除的分支 需要进行合并的分支, 先包含需要合并的分支后, 在进行去除操作

贮藏和清理

要切换分支, 但当前分支的内容还没有保存, 又不想进行一次提交: git stash [push] [-m 暂存注释], push可以省略

应用某次贮藏: git stash (apply|pop) [–index] [stash_id], 会将贮藏的文件恢复到工作区, 也包括以前暂存了的文件. apply只会应用贮藏, 而pop应用后, 会删除. –index, 以前在暂存区的还原到暂存区, 以前在工作区的还原到工作区. 不写stash_id, 默认最近一次贮藏

显示贮藏列表: git stash list

删除某次贮藏: git stash drop stash_id

清空全部贮藏: git stash clear

从贮藏中创建一个分支: git stash branch 新分支名 [stash_id], 会切换到该分支, 也会删除原贮藏

实用的git命令

选择文件暂存, 选择文件跟踪, 暂存一个文件的部分修改: git add -i, 可以进入交互式暂存

从工作区移除未被忽略文件忽略的同时又没有被跟踪的文件: git clean -i [-n]. -n, 如果执行, 会被移除的文件, 这次并没有执行

恢复到某个先前版本, 将文件恢复到某次提交, 将分支和head移动到某次提交: git reset [提交的hash值] [路径名 [–patch]] (–soft|–mixed|–hard). 不写hash值, 默认head指定的提交; 如果指定路径名, 则只会针对该文件; –patch, 选择部分恢复; –soft, 仅移动分支和head的指向; –mixed, 还会将暂存区恢复成这次提交, 也是默认选项; –hard, 前两个的功能加上将工作区恢复成该次提交. git reset –hard和git checkout的区别: 前者会移动HEAD指向, 后者只会移动HEAD自身; 前者对工作区不安全, 强制覆盖, 后者安全, 如果当前工作环境不为空, 不会允许切换

将某次提交后的所有提交压缩成一次提交: git reset 该次提交的hash值 –soft; 然后git 提交一次. 本质是利用暂存区里的文件, 进行一次提交

撤销合并: git reset [“head^”|head] –hard. head默认选择第1父分支

对某次提交之后的提交批量重命名: git rebase -i 该次提交; 将pick修改为edit, 退出; git commit –amend; 在vi中修改注释; git –continue. 如果有多个edit, 重复运行后三步. rebase不要在远程仓库运行, 也不要在已经上传到远程仓库的提交运行, 只在本地仓库还没有向远程仓库push的提交运行
